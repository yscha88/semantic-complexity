# 실험 설계 프로토콜

> 모든 실험은 이 프로토콜을 따라야 한다. 따르지 않은 실험은 무효.

---

## 1. 실험 전 체크리스트

```
□ 독립변인이 정확히 1개만 변하는가?
□ 나머지 변인(코드 길이, 스펙 복잡도, 프롬프트 형식)이 모두 동일한가?
□ 주석/힌트가 답을 알려주지 않는가?
□ 인공 코드가 아니라 실제 오픈소스 코드인가?
□ 제3자가 동일 조건에서 재현 가능한가? (커밋 해시, 프롬프트 전문 기록)
□ 인용한 이론의 범위를 초과하여 주장하지 않는가?
□ 이전 실험의 교란 변인이 반복되지 않는가?
```

**하나라도 미충족 시 실험을 시작하지 않는다.**

---

## 2. 실험 설계 양식

모든 실험은 실행 전에 다음 양식을 작성한다:

```markdown
### EXP-XX: [제목]

**가설**: [반증 가능한 형태로 서술]

**독립변인**: [정확히 1개]
**종속변인**: [측정 방법 포함]
**통제변인**: [동일하게 유지하는 모든 것]

**교란 변인 점검**:
- [변인 1]: [어떻게 통제하는가]
- [변인 2]: [어떻게 통제하는가]

**대상 코드**:
- 레포: [URL]
- 커밋: [해시]
- 파일: [경로]
- 라이선스: [MIT/BSD/...]

**프롬프트**: [전문 기록]

**모델**: [카테고리 + 횟수]

**성공 기준**: [사전 정의, 실험 후 변경 금지]
**실패 기준**: [사전 정의]
```

---

## 3. 금지 사항

| 금지 | 이유 | 실패 사례 |
|------|------|----------|
| 주석/힌트로 답 제공 | LLM이 주석을 읽고 "찾은 것"처럼 보임 | EXP-04a `# BUG: should be >= 10` |
| 인공 코드로 개념 수 실험 | 개념 수와 스펙 복잡도가 동시에 변함 | EXP-04d V7 vs V15 |
| 실험 후 채점 기준 변경 | 결론에 맞추어 기준을 조정하게 됨 | — |
| 비공개 코드 결과를 공개 근거로 사용 | 제3자 재현 불가 | EXP-01v (inference-api, 삭제됨) |
| 이론의 범위를 초과하여 인용 | 학술적 부정직 | Cousot 1977을 SAR 근거로 (수정됨) |
| "세션이 극한"이라며 대충 넘기기 | 검증 없는 결과가 축적됨 | EXP-04 전반 |

## 4. 실제 코드 사용 원칙

인공 코드(synthetic code)는 **예비 관찰(pilot)**에만 사용하며,
Tier 2 승격 근거로 인정하지 않는다.

**실제 코드 기준**:
- 공개 오픈소스 레포에서 가져온 코드
- 커밋 해시로 특정 버전 고정
- 제3자가 클론하여 동일 코드를 확인 가능

**실제 버그 기준**:
- fix 커밋이 존재하는 실제 결함
- 또는 BugsInPy 같은 검증된 버그 데이터셋
- 인공 버그 주입(fault injection)은 주입 사실을 명시하고 pilot으로만

## 5. 변인 통제 가이드

### 개념 수 실험
```
잘못된 설계: V7(7개 개념, 스펙 5줄) vs V15(15개 개념, 스펙 12줄)
→ 개념 수와 스펙 복잡도가 동시에 변함

올바른 설계: 동일 기능의 코드를 함수 분리만 다르게
→ A(단일 함수, 개념 15) vs B(분리된 함수, 각 개념 5)
→ 기능, 줄 수, 스펙 모두 동일, 구조만 다름
```

### SKILL 효과 실험
```
잘못된 설계: A(코드만) vs B(코드+SKILL)
→ B의 프롬프트가 더 길어서 "더 많은 정보" 효과와 분리 불가

올바른 설계: A(코드+일반 지시) vs B(코드+SKILL 규칙)
→ 프롬프트 길이를 동일하게 맞춤
→ 또는 A에도 동일 길이의 무관한 텍스트 패딩
```

### 보안 분석 실험
```
잘못된 설계: "보안 위험을 찾아라" vs "B1-B4로 분석하라"
→ B에 더 구체적인 지시 = 당연히 더 잘함

올바른 설계: 양쪽 모두 동일한 수준의 구체적 지시
→ A: "인증, 시크릿, 경계, injection을 분석하라"
→ B: "B1 인증, B2 시크릿, B3 경계, B4 injection을 분석하라"
→ 차이: 내용이 아니라 "프레임의 존재 여부"만
```

## 6. 결과 기록 양식

```markdown
### 결과

**체크리스트 충족**: 7/7 (또는 N/7 — 미충족 항목 명시)

**데이터**:
| 조건 | 모델 | 결과 | 비고 |
|------|------|------|------|

**관찰된 것** (사실만):
- 

**관찰되지 않은 것** (주장하면 안 되는 것):
-

**교란 변인 사후 점검**:
- [발견된 교란 변인이 있으면 기록]

**판정**: 승격 / 보류 / 무효
**무효 사유**: [있으면 기록]
```

## 7. 무효화 기준

실험 실행 후라도 다음에 해당하면 무효:

| 조건 | 예시 |
|------|------|
| 사후에 교란 변인 발견 | EXP-04: 주석 힌트 발견 |
| 체크리스트 미충족이 사후 확인 | 인공 코드 사용 |
| 결과에 맞추어 기준 변경 | — |
| 재현 불가 (커밋 해시 미기록 등) | — |

무효화된 실험은 RESEARCH.md에 **"무효 — 사유: [...]"**로 기록하고,
Tier 2 근거에서 제거한다.
