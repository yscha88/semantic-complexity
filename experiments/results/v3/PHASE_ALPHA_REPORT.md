## 1. 목적

"SKILL(구체적 판정 기준을 포함한 가이드)이 LLM 코드 분석의 품질을 높이는가?"를 검증한다.

## 2. 실험 설계

### 2.1 그룹 정의 — 가이드 구체성의 3단계

| 그룹 | 프롬프트 구조 | 예시 (보안 축) |
|------|-------------|--------------|
| **A** | 간략 지시 | "보안을 분석하라" |
| **B** | 체크리스트 (항목 나열) | A + "입력 검증, 인증, 시크릿, 민감 데이터를 확인하라" |
| **C** | SKILL (판정 기준 + 출력 형식) | A + B1~B4 규칙별 판정 기준 + pass/warning/fail + 수정 권고 |
| **D** | SKILL + 자율 탐색 | C + "체크리스트 완료 후 체크리스트 외 이슈도 식별하라" |

A~D는 동일 지시("보안을 분석하라")로 시작하며, 가이드 구체성만 단계적으로 증가한다.

## 3. 결과

### 3.1 SKILL 효과 — 🍞 보안 축 (R1/R2/R3, 22단위)

#### 3.1.1 그룹별 성과

| 그룹 | TP 평균 | 정밀도 평균 | 구조화 (0-2) |
|------|---------|-----------|-------------|
| A (간략) | 5.0 | 0.77 | 1 |
| B (체크리스트) | 5.8 | 0.79 | 1 |
| C (SKILL) | 2.75 | 0.75 | **2** |
| D (SKILL+자율) | **7.7** | 0.93 | **2** |

#### 3.1.2 관찰

**C(SKILL만)가 A보다 TP가 낮은 이유**: 체크리스트가 LLM에게 "완료 조건"을 부여한다. B1-B4를 전부 판정하면 "할 일이 끝났다"고 판단하고 탐색을 종료한다.

**이것이 토큰 한계 때문인지 검증**: max_completion_tokens를 4096→8192로 두 배 늘려도 출력 토큰이 변하지 않음(GPT: 1494→1540, Sonnet: 678→678). **토큰 여유가 있어도 더 쓰지 않는다.** 원인은 토큰 한계가 아니라 "task 완료 판정"이다.

**D(자율 탐색 추가)로 해소**: "After completing the checklist, perform an additional free-form analysis" 한 줄 추가 시, C에서 놓쳤던 TOCTOU, JWT 시크릿, 타이밍 공격 등을 추가로 탐지하면서 B1-B4 구조화도 유지.

#### 3.1.3 B vs D 직접 비교 (R3, 283줄)

| 이슈 | B에서 탐지 | D에서 탐지 | D가 추가로 찾은 이유 |
|------|----------|----------|-------------------|
| AUTH_FLOW 미선언 | 0/2 | **2/2** | B2 판정 기준이 명시적으로 요구 |
| request_2sa 입력 미검증 | 0/2 | **2/2** | B1 "진입점마다 검증" 기준이 구체적 함수 수준 검증 유도 |
| request_2fa_web payload 미검증 | 0/2 | **2/2** | 동일 |
| status_exchange 경쟁 조건 | 0/2 | **1/2** | 자율 탐색 구간에서 발견 |

B는 "입력 검증"을 항목으로 나열했지만, 어떤 함수의 어떤 진입점을 볼지 구체적으로 가이드하지 않았다. D는 판정 기준("진입점마다 검증 존재?")이 탐지를 함수 수준까지 가이드했다.

#### 3.1.4 SKILL 양방향 제어

같은 메커니즘이 반대로도 작동한다. A(간략)는 GPT-5.4에서 9개 이슈를 보고했으나 FP 2개(토큰 응답 노출, 이메일 인증 없음)가 포함됐다. SKILL로 범위를 한정하면 불필요한 이슈 보고가 줄고 해당 토큰이 관련 이슈 분석에 재배분된다.

| 방향 | SKILL 전략 | 효과 |
|------|-----------|------|
| 넓게 찾아야 할 때 | 체크리스트 + 자율 탐색 | 탐지 범위 확보 |
| 좁게 집중해야 할 때 | 체크리스트만 (자율 탐색 제외) | 토큰 절약, 오탐 감소 |

### 3.2 코드 구조 효과 — 과제 유형에 따라 다름 (8단위)

#### 3.2.1 분석 과제 (httpie before/after)

| | before (단일 404줄) | after (4모듈 460줄) |
|---|---|---|
| TP | 6.0 | 6.0 |
| 정밀도 | 0.92 | 0.92 |

**차이 없음.** httpie의 리팩토링은 파일 분리만 했고 로직은 동일. 같은 로직이면 같은 이슈를 찾는다.

#### 3.2.2 수정/배치 과제 (Django before/after)

| | before (handle() 단일 263줄) | after (load_label+process_dir 추출 270줄) |
|---|---|---|
| 배치 전략 | **근접 배치** — 코드 흐름 기반 ("line 194 뒤") | **의미적 배치** — 함수명 기반 ("process_dir() 안") |
| 양 모델 일치 | ✅ | ✅ |

Django 선정 이유: 파일 분리가 아닌 **함수 추출** 리팩토링, 줄 수 거의 동일(263→270), BSD 라이선스, 87K스타, 커밋 메시지 "Refactor loaddata for readability".

이전 세션의 EXP-02(openpilot)에서도 동일 패턴이 관찰됐으며, 이번 실험에서 **다른 레포, 통제된 모델로 재현**됐다.

### 3.3 CC 추정 능력 — 🧀 Cheese (R3, 4단위)

radon으로 실측한 cyclomatic complexity와 LLM 추정값을 대조했다.

| 함수 | radon 실측 | GPT-5.4 추정 | Sonnet 추정 |
|------|----------|-------------|------------|
| request_2fa | **22 (D)** | 15-18 (과소) | 26+ (과대) |
| request_2fa_web | **8 (B)** | 8-10 ✅ | 8 ✅ |
| authenticator | **7 (B)** | 6-8 ✅ | 7 ✅ |
| prompt_int_range | **5 (A)** | 4 ✅ | 4 ✅ |
| 단순 함수 6개 | **1-2 (A)** | 1-2 ✅ | — |

- **수치 정확도**: ±30% — 정밀 측정 불가
- **방향성 판정** ("이 함수가 복잡한가?"): **10/10 일치** — 3함수(복잡) + 7함수(단순) 전부 맞춤
- **B vs D**: CC 추정 정확도는 동등. D는 A/B/C/D/F 등급 기준을 제공하여 판정을 구조화

## 4. 결론

### 4.1 검증된 것

| # | 결론 | 근거 | 재현 |
|---|------|------|------|
| 1 | SKILL 체크리스트만 주면 LLM은 task 완료로 판정하고 탐색을 조기 종료한다 | C < A in TP, max_tokens 증가해도 동일 | R1, R2 × 2모델 |
| 2 | 자율 탐색 조건 추가로 시야 축소가 해소된다 | D ≥ A in TP, 구조화 유지 | R1, R2 × 2모델 |
| 3 | D(SKILL+자율) > B(체크리스트) — 판정 기준이 탐지를 가이드한다 | R3: B 7.5 → D 10.0 (+33%) | 3축 × 2모델 |
| 4 | 코드 구조는 분석 과제에 영향 없고, 수정 과제에 의미적 배치를 유도한다 | httpie TP 동일, Django 배치 전략 변화 | 2레포 × 2모델 |
| 5 | LLM은 CC 방향성을 정확히 판단하나 수치는 ±30% 오차 | radon 대조: 10/10 방향 일치, 수치 부정확 | 1레포 × 2모델 |
