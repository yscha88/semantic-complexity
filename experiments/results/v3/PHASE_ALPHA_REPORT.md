## 실험 1: SKILL의 구체성과 분석 품질

**설계**: 동일 코드에 동일 지시("보안을 분석하라"), 가이드 구체성만 변화.

| 그룹 | 내용 |
|------|------|
| B | 체크리스트 항목 나열 ("입력 검증, 인증, 시크릿, 민감 데이터를 확인하라") |
| D | 규칙별 판정 기준(B1~B4) + 출력 형식 + "체크리스트 외 이슈도 식별하라" |

**대상**: 3개 MIT 레포 (fastapi-realworld 93줄, BlackSheep JWT 217줄, icloud-photos-downloader 283줄)

**결과**:

| | B (체크리스트) | D (SKILL+자율) |
|---|---|---|
| TP 평균 | 5.8 | **7.7** (+33%) |
| 정밀도 평균 | 0.79 | 0.93 |
| 구조화 (0-2) | 1 | **2** |

D가 B보다 추가로 찾은 것 (3레포 공통):
- 판정 기준이 구체적 함수 수준 검증을 유도 (예: "진입점마다 검증 존재?" → request_2sa 미검증 탐지)
- 자율 탐색 구간에서 체크리스트 밖 이슈 탐지 (경쟁 조건, 상태 전이 등)

**재현**: 3레포 × 2모델에서 D > B 방향 일관.

## 실험 2: 체크리스트가 시야를 좁히는 현상

**발견 경위**: SKILL(판정 기준만, 자율 탐색 없음)을 적용하면 간략 가이드보다 TP가 낮았다.

**원인 검증**:

| 가설 | 테스트 | 결과 |
|------|--------|------|
| 출력 토큰 한계 | max_tokens 4096→8192 | 출력 토큰 변화 없음. **기각** |
| 체크리스트가 "완료 조건"을 만듦 | 자율 탐색 한 줄 추가 | TP 회복. **지지** |

**결론**: 체크리스트형 가이드는 LLM에게 "할 일 완료" 신호를 만든다. 출력 토큰 양과 무관하다. "체크리스트 완료 후 자율 탐색" 지시로 해소된다. 반대로, 범위를 좁히고 싶으면 체크리스트만 주면 토큰을 절약할 수 있다.

## 실험 3: 코드 구조가 LLM 작업에 미치는 영향

**설계**: 동일 로직의 before(스파게티)/after(리팩토링) 코드에 동일 프롬프트.

| 과제 | 코드 | 결과 |
|------|------|------|
| 보안 분석 (이슈 탐지) | httpie nested_json (파일 분리) | **차이 없음** — before TP 6.0 = after TP 6.0 |
| 기능 추가 (배치 판단) | Django loaddata (함수 추출) | **차이 있음** — before: 근접 배치, after: 의미적 배치 |

분석 과제에서는 로직이 지배적이라 구조 차이가 영향 없음.
수정 과제에서는 함수명이 "어디에 넣을지"를 가이드 — 2레포 × 2모델에서 재현.

## 실험 4: LLM의 cyclomatic complexity 추정

**설계**: icloud-photos-downloader 10개 함수에 대해 LLM 추정값과 radon 실측값 대조.

| | 결과 |
|---|---|
| 수치 정확도 | ±30% — 정밀 측정 불가 |
| 방향성 ("이 함수가 복잡한가?") | **10/10 일치** |

LLM은 "어떤 함수가 복잡한가"를 판단할 수 있으나, 정확한 수치는 정적 분석 도구가 필요하다.
