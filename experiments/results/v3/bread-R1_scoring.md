## 관찰

### 1. SKILL(C)이 TP를 줄였다
- A/B: TP 4-5개 → C: TP 2개 (양 모델 동일)
- **C는 B1-B4 프레임에 집중하면서 TOCTOU(I2), JWT 시크릿(I5), 감사 로깅(I10) 등을 놓침**
- 이전 αv2에서 관찰한 "시야 축소" 현상이 재현됨

### 2. B(체크리스트)가 가장 높은 TP를 보임 (GPT-5.4)
- GPT-5.4: A(4) < B(5) > C(2)
- Sonnet: A(4) = B(4) > C(2)
- 체크리스트가 "뭘 봐야 하는지" 환기하면서도 시야를 좁히지 않음

### 3. C는 구조화만 높이고 탐지력은 낮춤
- 구조화 점수: A/B(1) < C(2) — 양 모델 동일
- 그러나 정밀도: C ≤ A (같거나 낮음)
- **"체계적으로 정리했지만 덜 찾았다"**

### 4. AUTH_FLOW는 체크리스트 전용 이슈
- I13은 C에서만 탐지 — SKILL의 B2에 "AUTH_FLOW 선언"이 있기 때문
- 이건 "SKILL 효과"가 아니라 "지시에 있으니까 언급한 것"

### 5. 모델 간 차이보다 그룹 간 차이가 크다
- GPT-5.4와 Sonnet이 동일 그룹에서 유사한 패턴
- A≈B > C 방향이 양 모델에서 재현

## 1차 판정

| 질문 | 결과 |
|------|------|
| SKILL(C)이 A보다 정밀도가 높은가? | **아니오** — 같거나 낮음 |
| SKILL(C)이 A보다 TP가 많은가? | **아니오** — 오히려 적음 (A:4 → C:2) |
| SKILL(C)이 A에서 놓친 실제 이슈를 찾는가? | **아니오** — C가 찾고 A가 못 찾은 TP 없음 |
| B(체크리스트)가 A보다 나은가? | **GPT-5.4에서 예 (4→5), Sonnet에서 동일** |
| 2모델에서 동일 방향인가? | **예** — A≥C, B≥C가 양 모델에서 일관 |

### PHASE-ALPHA_design.md §10 성공 기준 대조

> "Group C의 정밀도가 Group A보다 높음" → **실패**
> "Group C의 구체성이 Group A보다 높음" → **실패** (같거나 낮음)
> "Group C가 A에서 놓친 실제 이슈를 탐지" → **실패**

**Phase α 결과: SKILL(상세 가이드)은 분석 품질을 높이지 않았다. 오히려 시야를 좁혀 탐지력을 낮췄다.**

## 추가 실험: 시야 축소의 원인 분석

### 가설 1: 출력 토큰 한계 — **기각**

max_completion_tokens를 4096→8192로 두 배 늘려도 출력 토큰이 변하지 않음.

| | max=4096 | max=8192 |
|---|---|---|
| GPT-5.4 C | 1,494 | 1,540 (+46) |
| Sonnet C | 678 | 678 (동일) |

토큰 여유가 있어도 더 쓰지 않았음. **출력 한계는 원인이 아님.**

### 가설 2: 체크리스트가 "조기 종료" 신호 — **지지됨**

SKILL 프롬프트에 한 줄 추가: "After completing the checklist, perform an additional free-form analysis: identify any security issues NOT covered by B1-B4 above."

| | C (SKILL만) | D (SKILL + 자율 탐색) | 차이 |
|---|---|---|---|
| GPT-5.4 토큰 | 1,494 | 2,044 | +550 |
| GPT-5.4 TP | 2 | 6+ | +4 |
| Sonnet 토큰 | 678 | 938 | +260 |
| Sonnet TP | 2 | 5+ | +3 |

D에서 B1-B4 판정은 그대로 유지하면서, 자율 탐색에서 TOCTOU, JWT 시크릿, 타이밍 공격, 비밀번호 해싱 등을 **추가로 탐지**함. 양 모델에서 동일 패턴.

### 결론: SKILL 작성 시 시야 축소 방지 규칙

**SKILL이 선별 조건(체크리스트)만 제시하면 LLM은 해당 task를 완료한 것으로 판정하고 탐색을 종료한다.** 이것은 출력 토큰의 양과 무관하다 — 토큰 여유가 있어도 "할 일이 끝났다"고 판단하면 멈춘다.

따라서:
- 체크리스트형 SKILL에는 **"체크리스트 완료 후 자율 탐색"** 조건을 반드시 포함해야 함
- 이 조건이 없으면 체크리스트 항목 외의 이슈를 체계적으로 놓침
- 이 조건이 있으면 체크리스트의 구조화 장점 + 자유 탐색의 탐지 범위를 모두 확보

이 발견은 SKILL 설계 원칙으로 반영해야 하며, 다른 레포/축에서 재현 여부를 확인해야 한다.

### 반대 방향: 범위 한정에 의한 토큰 절약

같은 메커니즘은 반대로도 작용한다. A(간략 가이드)는 GPT-5.4에서 2,188토큰을 소비하며 9개 이슈를 보고했으나, 그 중 FP 2개(I8 토큰 응답 노출, I11 이메일 인증)는 해당 코드의 취약점이 아닌 일반론이다.

SKILL로 분석 범위를 명시적으로 한정하면:
- 관련 없는 이슈의 보고를 줄이고 (오탐 감소)
- 해당 토큰을 관련 이슈의 상세 분석에 재배분할 수 있다 (정밀도 향상)

| 방향 | 문제 | SKILL 전략 | 효과 |
|------|------|-----------|------|
| 넓게 찾아야 할 때 | 체크리스트가 시야를 좁힘 | 자율 탐색 조건 추가 | 탐지 범위 확보 |
| 좁게 집중해야 할 때 | 무관한 이슈까지 보고 (토큰 낭비, 오탐) | 범위를 명시적으로 한정 | 토큰 절약, 정밀도 향상 |

**SKILL은 LLM의 주의 자원을 양방향으로 제어하는 도구다** — 확장도, 축소도 가능.
