## 분석

### 테스트 비율 (🥓 Ham H1)

| 티어 | 평균 테스트% | Gate 기준 |
|------|-----------|----------|
| 하위 (PoC) | 14.1% | PoC: ≥50% → **미달** |
| 중위 (MVP) | 17.4% | MVP: ≥80% → **미달** |
| 상위 (Prod) | 24.9% | Prod: ≥95% → **미달** |

**관찰**: 모든 티어가 Gate 기준에 미달. 오픈소스의 테스트 비율은 Gate가 요구하는 수준보다 훨씬 낮음.
**해석**: Gate 임계값(50/80/95%)은 "바람직한 수준"이지 "현실 수준"이 아님. 이는 의도적 — Gate는 현실이 아니라 목표를 설정하는 것.

### 테스트 0% 분포

| 티어 | 테스트 0% 레포 수 | 비율 |
|------|----------------|------|
| 하위 | 3/6 | **50%** |
| 중위 | 1/6 | **17%** |
| 상위 | 0/5 | **0%** |

**관찰**: 티어가 올라갈수록 테스트 0%가 줄어듦. **단조 감소** (50% → 17% → 0%).
**해석**: PoC Gate의 "돌아가면 OK" (테스트 없어도 됨)은 하위 레포의 현실과 일치.

| 티어 | 평균 최대 함수 | 추세 |
|------|-------------|------|
| 하위 | 133줄 | — |
| 중위 | 192줄 | ↑ |

**관찰**: 상위 레포일수록 더 큰 함수가 존재. **단조 증가**.
**해석**: 프로젝트가 성장하면서 함수가 비대해지는 기술부채 축적. Gate 시스템이 이를 제어하려는 목적과 일치.

**해석**: 성숙한 프로젝트일수록 보안을 더 많이 다룸. Gate의 "MVP부터 trust boundary 필수"와 방향 일치.

## 결론

| 지표 | 티어 상관 | Gate 방향 일치 |
|------|---------|-------------|
| 테스트 0% 비율 | ✅ 단조 감소 (50→17→0%) | ✅ PoC "OK" → Prod "필수" |
| 최대 함수 줄 수 | ✅ 단조 증가 (133→192→268) | ✅ 기술부채 축적, Gate가 제어 필요 |
| 인증 코드 양 | ✅ 단조 증가 (2.5→9→15.3) | ✅ 성숙도↑ → 보안↑ |
| 테스트 비율 평균 | ⚠️ 증가하지만 Gate 미달 | ⚠️ Gate는 "목표"이지 "현실"이 아님 |

**17개 레포에서 Gate 시스템의 방향성은 경험적으로 지지됨.**
**단, Gate 임계값(50/80/95%)은 현실 오픈소스보다 높음 — 이는 의도된 "목표 설정".**

## Gate 임계값은 도메인 프로필이어야 한다

현재 THEORY.ko.md의 Gate 임계값(테스트 50/80/95%)은 **단일 기준**으로 설정되어 있다.
그러나 SURVEY 결과와 실무 관찰에 따르면 **도메인별로 요구 수준이 근본적으로 다르다.**

### 도메인별 Gate 프로필 (제안)

| 도메인 | PoC 테스트% | MVP 테스트% | Prod 테스트% | 근거 |
|--------|-----------|-----------|-----------|------|
| **오픈소스 라이브러리** | ≥10% | ≥30% | ≥60% | SURVEY 현실: 상위 평균 25% |
| **상용 SaaS** | ≥30% | ≥60% | ≥80% | 업계 표준 |
| **의료기기 (IEC 62304)** | ≥60% | ≥90% | ≥100% (MC/DC) | Class C 소프트웨어: 수정 조건/판정 커버리지 필수 |
| **자동차 (ISO 26262)** | ≥50% | ≥80% | ≥100% (MC/DC) | ASIL-D: 구조적 커버리지 필수 |
| **금융 (SOX/PCI-DSS)** | ≥40% | ≥70% | ≥90% | 감사 추적 + 회귀 테스트 필수 |

### 🍞 Bread도 도메인별 차등

| 도메인 | PoC auth | MVP auth | Prod auth | 근거 |
|--------|---------|---------|---------|------|
| **오픈소스 라이브러리** | 없어도 됨 | 선택 | 선택 | 라이브러리는 호출자가 인증 |
| **상용 SaaS** | 없어도 됨 | 필수 | 필수+MFA | 사용자 데이터 보호 |
| **의료기기** | 필수 | 필수+감사로그 | 필수+RBAC+21CFR11 | FDA 규제 |
| **자동차** | 없어도 됨 | 필수 | 필수+HSM | 안전 필수 시스템 |

### 의미

Gate 시스템의 **구조**(PoC→MVP→Prod 점진적 강화)는 보편적이지만,
**임계값**은 도메인별로 조정 가능한 파라미터이다.

이는 THEORY.ko.md에서 이미 정의한 "프레임워크(수학적 구조) + 파라미터(경험적)" 분리와 일치:
- **구조**: Gate 3단계 + 점진적 강화 = 고정
- **파라미터**: 각 Gate의 임계값 = 도메인별 프로필로 조정

### SKILL.md 반영 방향

```yaml
# SKILL.md에 도메인 프로필 선택 메커니즘
domain_profile: "medical"  # 또는 "saas", "oss-lib", "automotive", "fintech"
# → 프로필에 따라 임계값 자동 적용
```

또는 `__domain_profile__` 마커를 코드에 선언하여 MCP가 자동 감지.
